# npm Lockfile and Install Discipline

> **Rule:** the lockfile (`package-lock.json`, `pnpm-lock.yaml`, or `yarn.lock`) is committed and reviewed alongside source. CI uses `npm ci` (or the equivalent `pnpm install --frozen-lockfile`, `yarn install --immutable`) — never plain `npm install`. The release gate runs `npm audit` against a tiered policy with an explicit, audited override path for accepted exceptions.

The premise is that *the code that runs in production is the code we vetted*. Without lockfile discipline, "the code we vetted" silently shifts as transitive dependencies update, version ranges resolve differently, and ad-hoc local installs drift from CI. Three disciplines close those gaps.

## Discipline 1 — Commit and review the lockfile

The lockfile records the exact version of every direct *and* transitive dependency. It is generated by the package manager, not hand-written, but it is committed because:

- It is what makes `npm ci` deterministic.
- It is what makes dependency review possible (a PR's lockfile diff is the visible record of every transitively-changed package).
- It is what makes vulnerability response possible (you can answer "do we use the affected version of X?" by reading one file).

Reviewing the lockfile diff on a dependency-changing PR is a real review step, not a checkbox. A reviewer should be able to answer:

- Which direct dependencies were added, removed, or version-changed?
- Which transitive dependencies came along?
- Are any of the changed packages on a watch-list (recently-compromised, low maintenance, known issues)?
- Did the version range shift in ways the developer didn't intend (a `^1.2.3` resolving to a new minor release)?

For large lockfile diffs, tooling like `npm-check-updates` outputs or PR-bot comments summarize what changed. The summary doesn't replace the diff; it makes the diff scannable.

## Discipline 2 — `npm ci` in CI; never `npm install`

The two commands are different operations:

- `npm install` — resolves the version ranges in `package.json`, may update the lockfile, may install different transitive versions than the lockfile records.
- `npm ci` — installs exactly what the lockfile says, fails if the lockfile and `package.json` are inconsistent, refuses to modify the lockfile.

In CI, `npm ci` is the only correct choice. The build environment must reproduce *the exact dependency tree the lockfile commits to* — anything else means CI is not testing what production will run.

In local development, `npm install` is fine when intentionally adding or updating dependencies. The expectation is that lockfile changes go into the next PR; running `npm install` and not committing the resulting lockfile is the source of "works on my machine" outcomes.

The same discipline applies to other package managers:

- **pnpm**: `pnpm install --frozen-lockfile` in CI.
- **Yarn (modern)**: `yarn install --immutable` in CI.

## Discipline 3 — Tiered audit policy with override

`npm audit` (or `pnpm audit`, `yarn npm audit`) reports known vulnerabilities in installed packages. The output is a tree of advisories with severity tags. Treating audit as a binary "fail on any finding" is impractical; treating it as a warning is theater. The middle path is a tiered policy:

| Severity | Default policy |
|---|---|
| Critical | Hard block. Cannot merge. Override requires named security-team approver. |
| High | Hard block by default. Override requires recorded reason and approver. |
| Moderate | Tracked finding. Allowed to land. Reviewed monthly with goal of resolution. |
| Low | Tracked finding. Allowed to land. No active resolution requirement. |
| Info | Not gated. |

The override path matters. Every override of a Critical or High finding is recorded with:

- The advisory ID.
- The reason override was granted (e.g. "no exposed surface uses the vulnerable code path"; "fix coming from upstream within 7 days; mitigated downstream").
- The approver.
- An expiration date — overrides should not be permanent.

Periodic review of the override list is a separate discipline; an override list that grows without ever shrinking is the warning sign.

## Lockfile-as-config

The lockfile is configuration that controls what runs in production. Per [[Engineering Philosophy/Principles/Configuration as Code Review]], this means:

- The lockfile is in version control.
- Changes go through review.
- A lockfile-changing PR is governed by the same review rigor as a code-changing PR.

A common anti-pattern: a "dependency update bot" PR that changes the lockfile and is rubber-stamped by a single approver. This is fine when the bot is well-tuned and the diff is genuinely routine; it is not fine when the bot is auto-merging without anyone reviewing the transitive-version changes.

## Pinning vs ranges

The `package.json` direct-dependency version ranges (`^1.2.3`, `~1.2.3`, `>=1.2.3 <2`, exact `1.2.3`) are an upstream choice. The lockfile pins regardless of what the range says. The two interact:

- **Conservative range (`~` or exact)** + **lockfile**: minimal transitive drift. The lockfile is rarely updated; updates are deliberate.
- **Permissive range (`^` or wider)** + **lockfile**: lockfile drifts every time it's regenerated. The lockfile makes any single CI run reproducible, but cross-CI drift is high.

For critical-path projects, conservative ranges + active lockfile maintenance is the safer combination. The cost is more PRs to update the lockfile; the benefit is the lockfile diff actually reflects what the developer asked to change.
